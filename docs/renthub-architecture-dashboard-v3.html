Auth Model — Two-Layer Design
UserRole — JWT Auth Layer
ADMINPlatform superuser. Accesses /admin/* routes.
USERAll non-admin users. Scoped further by UserType.
UserType — App Routing Layer
SYSTEM_ADMINPaired with ADMIN · controls platform
LANDLORDPaired with USER · manages properties
TENANTPaired with USER · views/pays bills
Login → JWT contains { role, userType } → RolesGuard checks role → UserTypeGuard checks userType → Client routes: SYSTEM_ADMIN→/admin · LANDLORD→/landlord · TENANT→/tenant

Raw SQL Migration Notes (Prisma limitations)
-- Must be added manually to migration SQL after prisma migrate dev:
-- 1. One active lease per unit (partial unique)
CREATE UNIQUE INDEX uq_lease_unit_active ON leases(unit_id) WHERE status = 'ACTIVE' AND deleted_at IS NULL;

-- 2. No duplicate active add-on per unit (NULLS NOT DISTINCT)
CREATE UNIQUE INDEX uq_unit_addon_active ON unit_addons(unit_id, addon_catalog_id, effective_until) NULLS NOT DISTINCT WHERE deleted_at IS NULL;

-- 3. Computed columns for meter readings
ALTER TABLE utility_readings ADD COLUMN units_consumed DECIMAL(12,4) GENERATED ALWAYS AS (current_reading - previous_reading) STORED,
ADD COLUMN amount_computed DECIMAL(12,2) GENERATED ALWAYS AS ((current_reading - previous_reading) * rate_per_unit) STORED;

-- 4. Rename existing Role enum
ALTER TYPE "Role" RENAME TO "user_role";

📋 API Conventions
List response: GET /resource?page=1&limit=20&sort=createdAt&order=desc → { data: [], total, page, limit }
Error shape: { error: { code: "VALIDATION_ERROR", message: "...", details?: [...] } }
Soft deletes: All GET endpoints exclude deleted_at IS NOT NULL rows automatically
Idempotency: Send Idempotency-Key: <uuid> header on payment POST to prevent double-charge
Webhook security: Verify PayMongo signature (X-PayMongo-Signature header) · return 200 immediately · process idempotently via transaction_ref

Scaling Phases
Phase 1 — MVP
  • Single API server + single DB instance
  • PgBouncer for connection pooling
  • Cron worker on same box or separate worker
  • Target: 10–50 concurrent users, <1 GB DB
Phase 2 — Growth
  • Multiple API instances behind load balancer
  • Redis for refresh token cache + rate limiting
  • Same DB with read replica for dashboards
  • Target: 100–500 concurrent users
Phase 3 — Scale
  • DB read replica active for reporting queries
  • Separate job/worker pool for billing + notifications
  • Queue (Redis / SQS) for async webhook processing
Phase 4 — Enterprise
  • Multi-region only if latency demands it
  • SLA definitions for paying enterprise customers
  • Disaster recovery & formal incident response
